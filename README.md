# Birzeit School Registration Prototype

A SwiftUI app we built for our Software Engineering course (COMP433, Birzeit University). It's a parent-facing portal for a single school — sign up, submit an application for your kid, pay registration and tuition through eSadad, and see grades and the class schedule once they're registered. There are two staff sides too: a school administrator who reviews applications, and a finance staff member who pulls up payment records.

Arabic-first with RTL layout, but the whole UI flips to English from the globe button in the header.

## What's in here

Everything in the four phase PDFs (requirements, use cases, sequence diagrams, architecture) is implemented as actual screens:

- **UC1 Create Parent Account** — National ID + mobile, verified through a fake "National Records Database" service (any 9-digit ID starting with a non-zero digit works), SMS OTP (`1234`), then auto-generated username + temporary password.
- **UC2 Log In** — single login screen, role detected from the credentials. Failed-attempt counter, 5-strike suspension, SMS credential recovery.
- **UC3 Submit Registration Application** — pick a grade (KG1 → Grade 12), pick an existing child or add a new one, review, submit. Duplicate-application guard.
- **UC4 View Application Status** — pending / accepted / rejected with the decision reason if rejected.
- **UC5 Pay Registration Fee** + **UC8 Pay Tuition Fee** unified under one **Make Payment** entry, which extends to an **Apply Multi-Child Discount** step (5% per additional registered child, capped at 10%).
- **eSadad flow** — we generate a unique reference number, ask the parent to enter it inside the eSadad app to look up the bill, and only mark the payment as paid once they confirm completion. Tuition supports full payment or a 4-month installment plan with a visible schedule.
- **UC6 Academic Info** — grades by subject, weekly class schedule, school-year calendar.
- **UC7 Switch Between Children** — selected-child context flows through to applications, payments, and academic.
- **UC9 Review Registration Application** (admin) — accept or reject with a required reason; the parent gets a real notification in their inbox bell.
- **UC10 View Payment Records** (finance) — search by student or ID, see transactions, paid amount, outstanding balance, next installment date, and the full installment schedule.

Persistence is local JSON in the app's Documents folder, so accounts and payments survive a relaunch.

## Stack

- SwiftUI, iOS 16+
- One Codable `DataRepository` as the source of truth (no backend — everything is in-process so you can run it standalone)
- Project file is generated by [XcodeGen](https://github.com/yonki/XcodeGen) from `project.yml` so the repo doesn't carry a giant pbxproj diff
- No third-party dependencies

## Run it

```bash
brew install xcodegen        # if you don't already have it
cd SchoolRegistration
xcodegen generate
open SchoolRegistration.xcodeproj
```

Pick an iPhone simulator and hit Run. App defaults to Arabic; toggle the globe in the top corner for English.


## Known limitations (be honest)

- The "send SMS within 2 minutes" SR is treated as immediate — we don't model a delivery SLA on the client side.
- The eSadad bank service is a local mock; nothing actually leaves the device.
- No real role-based service layer, since the whole thing is a single-process iOS client — the role gate is enforced where the session is consumed.
- Sequence diagrams in the phase docs assume a backend (web layer → access control → services → adapters); this implementation collapses those into the `DataRepository` + service-style enums for the demo.
